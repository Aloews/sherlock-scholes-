# Методология: построение двуязычной (RU/EN) базы футболистов топ-5 лиг для Telegram Mini App «Шерлок Скоулс»

## TL;DR
- **Рабочее ядро — две независимые связки:** (1) **API-Football (api-sports.io)** для составов и статистики игроков по 5 лигам × сезонам, и (2) **Wikidata + русская Wikipedia** для получения русских имён. Ранжирование «топ-100 популярных» строится как взвешенная композиция трёх прокси-сигналов: просмотры страниц Wikipedia (Wikimedia Pageviews API), трансферная стоимость (Transfermarkt) и доля сыгранных минут (API-Football).
- **Русские имена надёжнее получать НЕ транслитерацией, а через цепочку** «английское название статьи Wikipedia → Wikidata Q-id → sitelink ruwiki», с дизамбигуацией однофамильцев через фильтр occupation P106 = Q937857. Транслитерация (библиотека `transliterate`) — только запасной путь с понижением флага confidence. Канонический пример, доказывающий необходимость этого подхода: Эрлинг Холанн пишется в русской Wikipedia как **«Холанн»**, а не «Холанд» и не «Холланд» — транслитерация здесь ошибётся, sitelink — нет.
- **Хранить в отдельной таблице `players_meta` + связующей `player_seasons`** (отношение многие-ко-многим), не трогая существующие 1482 карточки в `cards`. Идемпотентность повторных запусков обеспечивается UNIQUE-ограничением и UPSERT (`ON CONFLICT`) по стабильному ключу (`wikidata_qid` / `api_football_id`), а не по автоинкрементному `id`.

## Key Findings

1. **football-data.org бесплатно не отдаёт данные по игрокам** — он годится для расписаний и таблиц, но не для нашей задачи: на free-тарифе нет составов и статистики игроков, а нужные данные продаются дорогими add-on'ами. Для сбора игроков это тупик.
2. **API-Football на free-тарифе (100 запросов/день) даёт ВСЕ эндпоинты, но ограничивает доступные сезоны.** Это значит, что исторические сезоны (особенно 2018–2020) на бесплатном плане, скорее всего, недоступны — для полного диапазона 2018–2025 почти наверняка потребуется платный план Pro ($19/мес, 7 500 запросов/день).
3. **«Популярность» прямой метрикой нигде не отдаётся — это всегда реконструкция по косвенным сигналам.** Для прошлых сезонов (например 2019/20) это оценка задним числом, а не точное измерение тогдашнего интереса. Это обязательно помечать флагом confidence.
4. **Ключевая техническая победа — сопоставление имён через Wikidata sitelinks, а не транслитерацию.** Один запрос `wbgetentities` с `sites=enwiki&sitefilter=ruwiki` решает 90%+ случаев точно; SPARQL-фильтр по P106 решает однофамильцев; транслитерация остаётся лишь для игроков без русской статьи.

## Details

### 1. Источники данных по игрокам, лигам и сезонам

**Топ-5 лиг и их идентификаторы в API-Football** (это и есть «коды лиг», которые надо зафиксировать в конфиге парсера):

| Лига | Код (наш) | API-Football League ID |
|---|---|---|
| Английская Премьер-лига | PL | 39 |
| Испанская Ла Лига | PD | 140 |
| Итальянская Серия A | SA | 135 |
| Немецкая Бундеслига | BL1 | 78 |
| Французская Лига 1 | FL1 | 61 |

**Сравнение API (актуально на 2025/2026):**

- **football-data.org** — free-тариф: ровно **10 запросов в минуту** и доступ к **12 турнирам** (Лига чемпионов, Премьер-лига, Ла Лига, Бундеслига, Серия A, Лига 1, Эредивизи, Примейра, Чемпионшип, бразильская Серия A, ЧМ ФИФА и чемпионат Европы). Однако на бесплатном тарифе **нет данных по игрокам** (составы, замены, карточки) и нет статистики матчей; глубокие данные продаются add-on'ами (livescores €12, deep data €29, statistics €15, odds €15 — в сумме €70+/мес). Проект ведёт Daniel Freitag с 2013 года. **Вывод: для сбора игроков по сезонам не подходит.**
- **API-Football (api-sports.io / api-football.com)** — free-тариф: **100 запросов/день**, 10 запросов/минуту, сброс квоты в 00:00 UTC. **Все эндпоинты доступны на free**, но free-план **ограничен по доступным сезонам** («Free plans are limited in terms of available seasons»). Платные планы: Pro $19/мес (7 500 запросов/день), Ultra $39, Mega $99. **Это основной источник игроков для проекта.**
- **Sportmonks** — free-план покрывает только **2 лиги** (датская Суперлига и шотландская Премьершип), для топ-5 непригоден. Платный Starter — €29/мес (€23 при годовой оплате) за 5 лиг на выбор. Исторические данные старше трёх сезонов — отдельная разовая покупка.
- **Transfermarkt** — официального публичного API **нет**; данные (рыночная стоимость, история трансферов) получают парсингом HTML или через сторонние обёртки (R-пакет `worldfootballR`, Apify-акторы, неофициальные `ceapi`-эндпоинты). Важная оговорка: **рыночная стоимость на Transfermarkt — это оценка сообщества, а не официальная сумма трансфера**, и обновляется она раз в несколько месяцев. Это серая зона с точки зрения ToS — действовать аккуратно, с rate-limit и User-Agent.

**Какие метрики реально доступны для определения «топ-100 популярных»:**

| Сигнал | Источник | Доступность |
|---|---|---|
| Минуты на поле, голы+пасы, рейтинг матчей | API-Football (`/players`, `/players/topscorers`) | Высокая (на платном плане) |
| Трансферная стоимость | Transfermarkt (парсинг) | Средняя (community-оценка) |
| Просмотры страницы Wikipedia | Wikimedia Pageviews API | Высокая, бесплатно |

Прямой метрики «популярность» ни один API не отдаёт — поэтому популярность всегда вычисляется как композит прокси-сигналов.

**Ключевые эндпоинты API-Football** (база `https://v3.football.api-sports.io/`, заголовок `x-apisports-key: <KEY>`):
- `GET /players?league=39&season=2021` — все игроки лиги-сезона. **Пагинация по 20 на страницу**: для Премьер-лиги это 500+ зарегистрированных игроков → 25+ последовательных вызовов. Всегда читать `paging.total` из первого ответа и идти в цикле.
- `GET /players/topscorers?league=39&season=2021` — топ-20 бомбардиров, каждый с полным статистическим профилем.
- `GET /players/squads?team=...` — текущий состав без сезонной статистики.

> **Бюджет вызовов** (важно при 100 req/день на free): 5 лиг × 7 сезонов × ~25 страниц = ~875 вызовов только на полный обход игроков. На free-плане это физически невозможно за один день и упирается в ограничение сезонов — закладывайте платный план Pro либо растягивайте сбор на несколько суток с кешированием.

### 2. Как ранжировать по популярности

**Wikimedia Pageviews API** (бесплатный, без ключа, данные с 1 июля 2015 — подтверждено первоисточником Wikimedia Analytics API: «These endpoints serve data starting on July 1, 2015»). Endpoint per-article:
```
https://wikimedia.org/api/rest_v1/metrics/pageviews/per-article/{project}/{access}/{agent}/{title}/{granularity}/{start}/{end}
```
Конкретный пример — просмотры русской статьи об Холанне за окно сезона 2022/23:
```
https://wikimedia.org/api/rest_v1/metrics/pageviews/per-article/ru.wikipedia.org/all-access/user/Холанн,_Эрлинг/monthly/20220801/20230601
```
Правила:
- Использовать агент **`user`** (а не `all-agents`) — это отфильтровывает ботов и краулеры.
- Title нужно URL-кодировать, пробелы → `_`.
- Для популярности «за сезон» суммировать просмотры за окно сезона (примерно август → май/июнь).
- Ответ: массив `items[]`, каждый с полем `views`.

**Композитный рейтинг популярности (рекомендуемая формула):**
1. Для каждой пары (лига, сезон) собрать три сигнала по всем кандидатам: `pageviews`, `market_value_eur`, `minutes_share`.
2. Нормализовать каждый сигнал **внутри этой лиги-сезона** (min-max в [0,1] или z-score) — нельзя сравнивать абсолютные просмотры между сезонами напрямую.
3. Взвесить и сложить. Стартовые веса: **Wikipedia pageviews 0.5, трансферная стоимость 0.3, доля сыгранных минут 0.2**. (Pageviews — самый прямой прокси известности у широкой аудитории, что и нужно для Alias-игры; минуты добавляют «реально игравших», а не только звёзд на скамейке.)
4. Отсортировать по `popularity_score` убыванию, взять топ-100 на лигу-сезон.

**Честное ограничение (обязательно задокументировать в проекте):** популярность игрока в прошлом сезоне (например 2019/20) — это **реконструкция задним числом**. Даже если брать pageviews за историческое окно тех лет, остаются искажения: русская статья могла быть создана позже сезона; название статьи могло измениться (переименование); ретроспективный всплеск интереса (трансфер, скандал) искажает картину «тогдашней» популярности. Поэтому каждая такая оценка помечается флагом `popularity_confidence`, и для исторических сезонов он ниже, чем для последнего завершённого.

### 3. Сопоставление английских и русских имён (ключевая часть)

Цель: от английского имени (латиница, как отдаёт API-Football) получить корректное русское написание из **русской Wikipedia**. Транслитерация «в лоб» ненадёжна — её используем только как fallback.

#### 3.1. Основной путь — один запрос Wikidata (без неоднозначности)

Если известно английское название статьи Wikipedia, можно за **один вызов** получить и Q-id, и русское название статьи, и русский label:
```
https://www.wikidata.org/w/api.php?action=wbgetentities&sites=enwiki&titles=Erling%20Haaland&props=sitelinks|labels&languages=ru&sitefilter=ruwiki&normalize=1&format=json
```
Разбор параметров:
- `sites=enwiki&titles=...` — ищем сущность по странице английской Wikipedia (так результат гарантированно относится именно к этой статье — неоднозначности однофамильцев на этом шаге нет, т.к. название enwiki-статьи само уже дизамбигуировано, например `Danny Rose (footballer, born 1990)`).
- `props=sitelinks|labels`, `languages=ru`, `sitefilter=ruwiki` — забираем только нужное: русский sitelink и русский label.
- `normalize=1` — чинит регистр и подчёркивания (работает только при одном site и одном title).

**JSON-пути для извлечения:**
- Q-id — единственный ключ под `entities` (например `entities.Q28967995`).
- Русское название статьи: `entities.<Qid>.sitelinks.ruwiki.title`
- Русский label: `entities.<Qid>.labels.ru.value`

#### 3.2. Альтернатива через pageprops (если стартуем с enwiki-страницы)

```
https://en.wikipedia.org/w/api.php?action=query&prop=pageprops&titles=Erling%20Haaland&format=json&redirects=1
```
Q-id = `query.pages.<pageid>.pageprops.wikibase_item`. **Параметр `redirects=1` обязателен** — иначе для имени-алиаса, которое редиректит на основную статью, поле вернётся пустым.

#### 3.3. Полный объект (для массовой обработки)

```
https://www.wikidata.org/wiki/Special:EntityData/Q28967995.json?flavor=simple
```
`?flavor=simple` отдаёт только truthy-statements + sitelinks + версию — меньше по объёму и хорошо кешируется. Путь: `entities.<Qid>.sitelinks.ruwiki.title`. Минус: возвращает ВСЕ языки без серверной фильтрации, поэтому для одной пары языков путь 3.1 экономнее.

#### 3.4. Дизамбигуация однофамильцев через SPARQL

Когда на руках только строка-имя (без чистого enwiki-title), `wbsearchentities` вернёт несколько кандидатов и **не умеет фильтровать по профессии**. Тогда отбираем футболистов SPARQL-запросом к `https://query.wikidata.org/sparql`:
```sparql
SELECT ?item ?enLabel ?ruArticleName WHERE {
  ?item wdt:P106 wd:Q937857 .          # occupation = association football player
  ?item wdt:P31  wd:Q5 .               # instance of = human
  ?item rdfs:label ?searchLabel .
  FILTER(LANG(?searchLabel) = "en")
  FILTER(CONTAINS(LCASE(?searchLabel), LCASE("Ronaldo")))
  OPTIONAL {
    ?ruArticle schema:about ?item ;
               schema:isPartOf <https://ru.wikipedia.org/> ;
               schema:name ?ruArticleName .
    FILTER NOT EXISTS { ?ruArticle wikibase:badge wd:Q70894304 . }  # исключить редиректы
  }
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en". ?item rdfs:label ?enLabel. }
}
```
Триплет `?ruArticle schema:about ?item ; schema:isPartOf <https://ru.wikipedia.org/> ; schema:name ?ruArticleName` отдаёт русское название статьи. Дополнительная дизамбигуация при множестве кандидатов — добавить условия по клубу (`wdt:P54`), году рождения (`wdt:P569`) или гражданству (`wdt:P27`), сверяя с данными из API-Football.

> **Производительный паттерн:** не гонять `CONTAINS` по всей базе. Сначала `wbsearchentities` → собрать кандидатов Q-id → затем SPARQL с блоком `VALUES ?item { wd:Q… wd:Q… }` плюс `?item wdt:P106 wd:Q937857`. Так быстрее и не упирается в таймауты.

#### 3.5. Подтверждённые идентификаторы Wikidata

- **P106** — occupation (профессия).
- **Q937857** — association football player; определение в Wikidata: «person who plays association football (soccer)».
- **P54** — member of sports team (членство в команде).
- **Q5** — human; **P31** — instance of.
- **Q70894304 / Q70893996** — бейджи sitelink-на-редирект (ручной/автоматический), для исключения «ложных» русских статей-редиректов.

#### 3.6. Проблемные случаи и как их решать

- **Холанд / Холланд / Холанн.** Русская Wikipedia использует написание **«Холанн, Эрлинг»** (статья: `Холанн,_Эрлинг`; в тексте: «Э́рлинг Брёут Хо́ланн (норв. Erling Braut Haaland, при рождении написание фамилии — Håland)»). Это эталонный кейс: транслитерация дала бы «Холанд» или ошибочную двойную «л» «Холланд» (которой в оригинале нет), а sitelink ruwiki даёт правильное «Холанн». **Вывод: всегда приоритет sitelink над транслитерацией.**
- **Однофамильцы** (несколько «Ronaldo», несколько носителей фамилии «Холанн» — Альф-Инге, Эрлинг и др.): отбор через P106 = Q937857 + сверка клуба/года рождения с данными API-Football.
- **Игрок без русской статьи в Wikipedia:** ключ `sitelinks.ruwiki` отсутствует (обрабатывать KeyError мягко). Топ-звёзды (Холанн, Месси, Роналду) русскую статью имеют почти всегда; молодёжь и игроки-ротация — часто нет. Тогда → fallback-транслитерация.
- **Баг wbsearchentities:** возвращает label на английском даже при `language=ru`. Русский label брать **только** из `sitelinks`/`labels.ru` через `wbgetentities` или `Special:EntityData`, не из `wbsearchentities`.

#### 3.7. Fallback-транслитерация

Когда русской статьи нет, использовать Python-библиотеку **`transliterate`** (PyPI, есть `ru` language pack) или собственную таблицу соответствий латиница→кириллица. Такие записи помечать `name_source = 'translit'`, `name_confidence = 'low'`, чтобы их можно было потом вручную выверить или скрыть из игры. Транслитерация хороша для приблизительного звучания, но систематически промахивается на нестандартных передачах (норвежские, нидерландские, славянские фамилии), поэтому она именно запасной путь.

### 4. Схема хранения данных (PostgreSQL / Supabase)

Принцип: **не трогаем существующую таблицу `cards` (1482 карточки).** Создаём две новые таблицы и опционально мягко связываем.

```sql
-- мета игроков (один игрок = одна строка, независимо от лиг/сезонов)
CREATE TABLE players_meta (
  id              BIGSERIAL PRIMARY KEY,
  wikidata_qid    TEXT UNIQUE,            -- стабильный внешний ключ
  api_football_id BIGINT UNIQUE,          -- стабильный ключ из API-Football
  name_en         TEXT NOT NULL,
  name_ru         TEXT,
  name_source     TEXT,                   -- 'wikidata' | 'translit'
  name_confidence TEXT,                   -- 'high' | 'low'
  created_at      TIMESTAMPTZ DEFAULT now(),
  updated_at      TIMESTAMPTZ DEFAULT now()
);

-- связь многие-ко-многим: игрок × лига × сезон + метрики популярности
CREATE TABLE player_seasons (
  id                    BIGSERIAL PRIMARY KEY,
  player_id             BIGINT REFERENCES players_meta(id),
  league                TEXT NOT NULL,    -- 'PL','PD','SA','BL1','FL1'
  season                INT  NOT NULL,    -- 2018..2024 (год начала сезона)
  popularity_rank       INT,             -- 1..100 в пределах лига-сезон
  popularity_score      NUMERIC,
  pageviews             BIGINT,
  market_value_eur      BIGINT,
  minutes_share         NUMERIC,
  play_level            NUMERIC,          -- метрика из существующего парсера
  popularity_confidence TEXT,            -- 'high'|'medium'|'low' — флаг косвенной оценки
  UNIQUE (player_id, league, season)      -- критично для идемпотентного UPSERT
);
```

Почему так:
- **Отношение многие-ко-многим** (игрок играл в разных лигах/сезонах) выносится в `player_seasons` — это нормальная форма, без дублирования имён.
- **Метрики популярности привязаны к (игрок, лига, сезон)**, а не к игроку, потому что популярность меняется по сезонам.
- **Флаг `popularity_confidence`** прямо в строке честно фиксирует, что это оценка по косвенным сигналам.

**Связь с существующей `cards`:** добавить в `cards` nullable-колонку и заполнять матчингом — это **не ломает** существующие 1482 строки (старые значения станут NULL):
```sql
ALTER TABLE cards ADD COLUMN player_meta_id BIGINT REFERENCES players_meta(id);
```
**Рекомендация: отдельная таблица `players_meta` + связь предпочтительнее, чем добавление множества колонок прямо в `cards`** — это изолирует новую логику, не раздувает игровую таблицу и позволяет пересобирать мету, не затрагивая карточки игры.

### 5. Процесс ежегодного обновления (идемпотентность)

Главный риск повторного запуска — дубликаты. Решение — **UNIQUE-ограничение + UPSERT (`ON CONFLICT`)**:

Через Supabase Python-клиент:
```python
supabase.table("player_seasons").upsert(
    rows,
    on_conflict="player_id,league,season"
).execute()
```
Чистый SQL-эквивалент:
```sql
INSERT INTO player_seasons (player_id, league, season, popularity_rank, popularity_score)
VALUES (...)
ON CONFLICT (player_id, league, season)
DO UPDATE SET popularity_rank = EXCLUDED.popularity_rank,
              popularity_score = EXCLUDED.popularity_score;
```

Критические правила (по реальным граблям Supabase):
- На колонки из `on_conflict` **обязан существовать UNIQUE-constraint или unique-index**, иначе PostgREST вернёт ошибку `42P10` («there is no unique or exclusion constraint matching the ON CONFLICT specification»).
- **Матчить и апсертить по стабильному внешнему ключу** (`wikidata_qid` или `api_football_id`), а НЕ по автоинкрементному `id`. Если делать upsert без стабильного ключа, Postgres может выдать новый `id` существующей строке — известный баг supabase-js, ломающий ссылки.
- Ежегодный прогон: добавляется новый сезон (например `season=2025`), парсер UPSERT'ит — старые сезоны не трогаются, новые добавляются, повторный запуск того же сезона перезаписывает, а не дублирует.

### 6. Практический пошаговый план для работы с Claude Code

1. **Зафиксировать конфиг:** 5 лиг с их League IDs (таблица выше) × 7 сезонов (2018–2024 как год начала). Проверить на free-плане API-Football, какие сезоны реально доступны; если исторические закрыты — заложить Pro-план ($19/мес).
2. **Создать схему** (`players_meta`, `player_seasons`, опциональная колонка в `cards`) — отдельной миграцией, не затрагивающей `cards`-данные.
3. **Тест на малой выборке:** прогнать парсер на ОДНОЙ лиге-сезоне (например PL 2022) и всего на 1 команде. Глазами проверить 10–20 русских имён (особенно «трудные»: Холанн, Дзюба-латиницей-обратно, нидерландцы, скандинавы). Только убедившись в качестве — масштабировать.
4. **Сначала наполнить `players_meta`** (имена EN/RU), потом `player_seasons` (метрики и ранги). Имя получать по приоритету: путь Wikidata sitelink (3.1/3.2) → при отсутствии fallback-транслитерация (3.7) с флагом.
5. **Посчитать `popularity_score`** по формуле из раздела 2, нормализуя внутри лиги-сезона, отсортировать, оставить **топ-100 на каждую лигу-сезон**.
6. **UPSERT в Supabase** по стабильным ключам (раздел 5).
7. **Контроль качества:** выборочно сверить топ-10 каждой лиги-сезона с интуицией (там должны быть звёзды), проверить долю записей с `name_confidence='low'` — если высока, доработать матчинг.

**Этичность и вежливость к источникам (подтверждаем критичность встроенного кеша):**
- **Wikimedia требует осмысленный User-Agent с контактом** (например `SherlockScholesBot/1.0 (you@example.com)`) — иначе запросы попадают в жёсткий rate-limit-класс для анонимных скрейперов, и в 2026 году это правило начали автоматически принудительно применять.
- К Wikimedia: **держать concurrency ≤ 1–3, задержку ≥ 1 секунды** между запросами, делать паузу ≥ 15 минут при 5xx.
- **Уважать `Retry-After` при 429/503**, реализовать exponential backoff.
- **Кеширование** (у вас уже встроено в `football_scraper`) — **самое важное** для соблюдения лимитов: исторические данные не меняются, поэтому повторно тянуть один и тот же сезон нельзя. Это подтверждает каждый гайд по этим API. Кешировать как ответы API-Football, так и резолвы Wikidata sitelinks (Q-id → ruwiki title).
- К Transfermarkt относиться особенно бережно (нет официального API, серая зона ToS): минимальная частота, User-Agent, по возможности избегать массового парсинга — рыночную стоимость можно тянуть только для уже отобранных топ-100, а не для всех подряд.

## Recommendations

**Этап 0 (настройка, до кода).** Зарегистрировать ключ API-Football; на free-плане проверить доступность сезонов 2018–2024. **Порог решения:** если доступны только последние ~3 сезона — переходить на Pro ($19/мес); если нужны все 7 сезонов сразу, Pro почти обязателен из-за бюджета вызовов (~875 запросов на полный обход против 100/день на free).

**Этап 1 (схема).** Применить миграцию с `players_meta` и `player_seasons`. **Не** добавлять колонки напрямую в `cards`, кроме одной nullable-ссылки `player_meta_id`. Проверить, что UNIQUE `(player_id, league, season)` создан — без него UPSERT не заработает.

**Этап 2 (пилот на 1 команде).** Собрать одну команду одного сезона, выверить русские имена вручную. **Порог:** если доля верных русских имён из sitelinks < 90% — сначала чинить матчинг (проверить `redirects=1`, `normalize=1`, дизамбигуацию P106), и только потом масштабировать.

**Этап 3 (полный сбор).** Лига за лигой, сезон за сезоном, с кешем и rate-limit. После каждой лиги-сезона — UPSERT и контрольная сверка топ-10.

**Этап 4 (ежегодное обновление).** Раз в год добавлять новый сезон тем же парсером — за счёт UPSERT по стабильному ключу старые данные не ломаются.

**Метрики/пороги для пересмотра подхода:**
- Если > 15–20% игроков топ-100 уходят в `name_source='translit'` — значит, охват ruwiki недостаточен; рассмотреть ручную выверку этих имён или второй проход дизамбигуации.
- Если веса композита дают «неузнаваемый» топ (в топ-10 нет очевидных звёзд) — увеличить вес pageviews и уменьшить вес минут.
- Если API-Football free упирается в лимит сезонов — порог перехода на Pro достигнут.

## Caveats

- **Популярность — всегда прокси-оценка.** Для прошлых сезонов это реконструкция задним числом, а не измерение тогдашнего интереса; помечать флагом `popularity_confidence`. Веса композита (0.5/0.3/0.2) — стартовая гипотеза, требующая калибровки на ваших данных.
- **Free-тарифы ограничены.** football-data.org бесплатно не отдаёт игроков вообще; API-Football free ограничивает сезоны; Sportmonks free не покрывает топ-5. Для полноценного исторического сбора почти наверняка нужен платный план API-Football.
- **Transfermarkt** — нет официального API, рыночная стоимость community-driven и нерегулярно обновляемая; парсинг — серая зона ToS, использовать минимально и вежливо.
- **Wikidata/Wikipedia — живые, изменяемые данные.** Sitelinks и названия статей могут меняться (переименования), статья может отсутствовать или быть редиректом (фильтровать бейджи Q70894304/Q70893996). SPARQL-сервис может отставать от живой вики на минуты; Action API и `Special:EntityData` — актуальные. Точная доля топ-игроков без ruwiki-статьи и частота коллизий имён в авторитетных источниках количественно не зафиксированы — это надо измерять на собственной выборке в пилоте (Этап 2).
- **Цены и лимиты API приведены по состоянию на 2025/2026** и могут измениться — перед оплатой свериться с актуальными прайсами провайдеров.
